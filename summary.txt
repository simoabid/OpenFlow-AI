1. PROJECT OVERVIEW
================================================================================

OpenFlow-AI is a revolutionary Android application that serves as a proactive,
on-device AI agent for Android. It acts as a personal phone operator that can:

- Understand natural language voice commands
- Autonomously navigate and operate phone UI
- Execute complex, multi-step tasks across different applications
- Remember user preferences and context (currently disabled)
- Respond with high-quality voice synthesis

TAGLINE: "You touch grass. I'll touch your glass."

PROJECT STATUS: Work In Progress (WIP)
LICENSE: Personal Use Only (Commercial license required for business use)
REPOSITORY: https://github.com/simoabid/OpenFlow-AI
PLAY STORE: Available on Google Play Store

2.1 MULTI-AGENT ARCHITECTURE
-----------------------------
OpenFlow-AI uses a sophisticated multi-agent system architecture:

┌─────────────────────────────────────────────────────────────┐
│                        USER                                 │
│                     (Voice/Text Input)                      │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                    AGENT SERVICE                            │
│              (v2.AgentService.kt)                          │
│   • Foreground Service hosting the AI Agent                │
│   • Manages agent lifecycle and task queue                 │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                   AGENT (v2.Agent.kt)                      │
│              Main Agent Loop: SENSE → THINK → ACT          │
│   ┌─────────────────────────────────────────────────┐     │
│   │  1. SENSE: Perception.analyze()                 │     │
│   │     - Capture screenshot or XML hierarchy       │     │
│   │     - Parse UI elements                         │     │
│   │  2. THINK: LLM Analysis                         │     │
│   │     - Build prompt with state & history         │     │
│   │     - Query Gemini AI for next action          │     │
│   │  3. ACT: ActionExecutor.execute()               │     │
│   │     - Tap, scroll, type, open apps              │     │
│   │     - File operations, web searches             │     │
│   └─────────────────────────────────────────────────┘     │
└────────────────────┬────────────────────────────────────────┘
                     │
        ┌────────────┼────────────┐
        │            │            │
        ▼            ▼            ▼
┌──────────┐  ┌──────────┐  ┌──────────┐
│  EYES    │  │  BRAIN   │  │  HANDS   │
│ (Perception)  (LLM)    │ (Actions)  │
│          │  │          │  │          │
│ Eyes.kt  │  │ GeminiAPI│  │ Finger.kt│
│ Screen   │  │ Gemini   │  │ Gestures │
│ Analysis │  │ AI Model │  │ Controls │
└──────────┘  └──────────┘  └──────────┘
        │            │            │
        └────────────┼────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│           ACCESSIBILITY SERVICE                             │
│       (ScreenInteractionService.kt)                        │
│   • Low-level access to screen hierarchy                   │
│   • Perform gestures (tap, swipe, long-press)             │
│   • Capture screenshots                                     │
│   • Input text and navigation (back, home, etc.)           │
└─────────────────────────────────────────────────────────────┘

2.2 KEY DESIGN PRINCIPLES
--------------------------
• Separation of Concerns: Eyes (perception), Brain (reasoning), Hands (action)
• Stateful Agent Loop: Maintains context across multiple steps
• Tool-Using AI: LLM decides which tools/actions to use
• Memory Management: Short-term (MemoryManager) and long-term (FileSystem)
• Error Handling: Retry mechanisms with exponential backoff
• Observable Execution: Real-time feedback through overlays and TTS

3. KEY FEATURES & CAPABILITIES

3.1 VOICE INTERACTION
---------------------
✓ Wake Word Detection:
  - "Hey OpenFlow-AI" using Porcupine engine
  - Alternative STT-based wake word detection
  - Automatic key management for Picovoice API
  
✓ Speech-to-Text (STT):
  - Press-and-hold voice command interface
  - Android's built-in SpeechRecognizer API
  - Multiple retry attempts for reliability
  - Visual feedback during recording
  
✓ Text-to-Speech (TTS):
  - High-quality voice using Google Cloud TTS (Chirp)
  - Local Android TTS fallback
  - Audio caching for common phrases
  - Debug mode controls
  - Thread-safe speech coordination

3.2 INTERACTIVE DIALOGUE SYSTEM
--------------------------------
✓ Automatic Clarification:
  - Detects unclear user instructions
  - Generates contextual clarifying questions
  - Progress tracking (e.g., "2 of 3 questions")
  - Dual input: voice + text responses
  
✓ ClarificationAgent (agents/ClarificationAgent.kt):
  - Analyzes instructions for ambiguity
  - Returns: CLEAR or NEEDS_CLARIFICATION
  - Dynamically generates question lists

Example Flow:
  User: "Message my brother happy birthday"
  System: Needs clarification
    Q1: "What is your brother's name in contacts?"
    Q2: "Which messaging app to use?"
  Enhanced: "Message my brother (John Smith) happy birthday via WhatsApp"

3.3 TRIGGER SYSTEM
------------------
✓ Multiple Trigger Types:
  1. SCHEDULED_TIME:
     - Precise time-based execution
     - Day-of-week selection
     - Uses AlarmManager with setExactAndAllowWhileIdle
     - Automatic rescheduling after boot
     
  2. NOTIFICATION:
     - Monitors notifications from specified apps
     - Executes tasks on notification receipt
     - OpenFlowNotificationListenerService
     
  3. CHARGING_STATE (ChargingStateReceiver):
     - Triggers when device charging state changes
     - Can execute tasks on plug-in/unplug

✓ Trigger Management:
  - TriggersActivity: View and manage all triggers
  - CreateTriggerActivity: Configure new triggers
  - TriggerManager: Persistence and scheduling
  - BootReceiver: Reschedule after reboot

3.4 TASK AUTOMATION
-------------------
✓ Supported Actions (v2/actions/Action.kt):

  - LongPressElement: Long press for context menus
  - InputText: Type text into fields

  - ScrollUp/ScrollDown: Vertical scrolling
  - OpenApp: Launch apps directly
  - SearchGoogle: Perform web searches

  - WriteFile/AppendFile/ReadFile: File operations

✓ UI Automation:

6. CORE COMPONENTS DEEP DIVE
================================================================================

6.1 AGENT V2 (v2/)
------------------
The version 2 architecture represents a complete rewrite with better separation
of concerns and more robust error handling.

Agent.kt (Main Loop):
• Entry Point: run(initialTask: String, maxSteps: Int)
• Loop Structure:
  1. SENSE: perception.analyze() → ScreenState
  2. THINK: 
     - memoryManager.createStateMessage() → Build prompt
     - llmApi.generateAgentOutput() → Get LLM decision
  3. ACT: 
     - actionExecutor.execute() → Perform actions
     - Record to history
  4. CHECK: 
     - If "done" action → exit
     - If max steps → exit
     - If consecutive failures >= maxFailures → exit
     - Else continue loop

AgentService.kt (Lifecycle):
• Foreground Service: Ensures long-running execution
• Task Queue: ConcurrentLinkedQueue for multiple tasks
• Notifications: Persistent notification while running
• Firebase Integration: Logs tasks to Firestore
• Stop Mechanism: Public stop() method and stop action intent

PromptBuilder.kt:
• System Prompt: Loaded from assets/prompts/system_prompt.md
• Dynamic Sections:
  - User Info
  - Agent History
  - User Request
  - Agent State (files, step, timestamp)
  - Android State (app, elements, screenshot)
  - Read State (transient data from last action)
• Action Signatures: Injected dynamically based on available actions

6.2 PERCEPTION SYSTEM
----------------------
Perception.kt:
• analyze() → ScreenState
• Two modes:
  1. XML Mode (default): Parse accessibility hierarchy
  2. Screenshot Mode: Capture visual representation
• SemanticParser: Extract interactive elements with IDs

Eyes.kt (Screen Capture):
• openEyes(): Bitmap screenshot via Accessibility Service
• openXMLEyes(): Formatted XML hierarchy
• openPureXMLEyes(): Raw XML
• getRawScreenData(): Combined data (XML + scroll info + dimensions)
• Efficiency: Single call for all data

SemanticParser.kt:
• Input: AccessibilityNodeInfo hierarchy
• Output: List of interactive elements with:
  - Numeric ID
  - Text description
  - Resource ID
  - State (clickable, enabled, focusable)
  - Type (widget class name)
  - Bounds (coordinates)
  - Hierarchy (indentation for parent-child)

6.3 ACTION SYSTEM
-----------------
Action.kt (Sealed Class):
• Type-safe action definitions
• Custom serializer for LLM JSON output
• Data-driven registry (Spec companion object)
• Snake_case for LLM compatibility

ActionExecutor.kt:
• execute(action: Action, screenState, context, fileSystem) → ActionResult
• Handles all action types:
  - UI: tap, long-press, scroll, input
  - Navigation: back, home, switch app
  - Apps: open app, launch intent
  - Communication: speak, ask
  - Files: read, write, append
